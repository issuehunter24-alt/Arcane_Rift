## 데이터 매핑 및 버전 관리 규칙

목표: `샘플_카드_세트_40.json` ↔ `web/public/data/cards.json` 간 일관성 유지와 변경 이력 추적.

### 1) ID/버전 정책
- 카드 ID는 전역 유일(`TYPE_THEME_INDEX`).
- `version` 숫자는 데이터 구조/수치가 바뀔 때 +1.
- 삭제 시 `replacementId` 맵을 문서 말미에 추가해 대체 카드 연결.

### 2) 필수/선택 필드
- 필수: id, name, type, rarity, cost, effects[], tags[], keywords[], version
- 선택: effectText, levelCurve{ base, perLevel }, vfxKey, sfxKey

### 3) 변환 파이프라인
1) 소스 편집: `샘플_카드_세트_40.json`
2) 변환/정규화: 스크립트로 `cards.json` 생성(순서, 타입 정합성 보장)
3) 검증: `npm run validate:data`로 스키마 및 일관성 점검

### 4) 변경 로그 포맷(예시)
```
2025-10-31 ATK_FIREBALL_003 v1→v2
- damage 30→32, Burn stacks 1 유지
- effectText "피해 32, 화상 1중첩" 갱신
사유: 초반 화력 부족 개선(기획서 수치 테이블)
```

### 5) 동기화 규칙
- `샘플_카드_세트_40.json`가 단일 소스. 수동 편집 금지: `cards.json`
- PR 체크리스트: 변환 스크립트 실행 결과 + 검증 통과 스크린샷 첨부

### 6) 릴리즈 태그
- `cards@YYYYMMDD` 태그로 데이터 스냅샷 생성(빌드 아티팩트와 동일 버전)
